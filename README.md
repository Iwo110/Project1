# Simple Chatbot Example

This project demonstrates a small conversational bot built on top of the
[HuggingFace Transformers](https://github.com/huggingface/transformers) library.
The bot keeps a history of the conversation in timestamped log files and can be
fine-tuned later on those logs.

## Setup

Install the required Python packages:

```bash
pip install -r requirements.txt
```

## Usage

Start an interactive chat session using the default DistilGPT2 model:

```bash
python main.py
```

To fine-tune the model on the conversation logs after chatting, run:

```bash
python -c "from chatbot import ChatBot; bot = ChatBot(); bot.fine_tune_on_logs()"
```

Fine-tuned models are saved in the `finetuned/` directory.

### Command line options

* `--model` - specify a different model name from HuggingFace.
* `--log-dir` - directory where chat logs will be written.
* `--fine-tune` - fine tune the model on all logs then exit.
* `--memory-file` - optional file used to persist chat history across sessions.

## Evolving Brain

The `brain.py` module builds on the basic `ChatBot` by adding a minimal "mood"
state and an evolution routine. Mood now responds to the sentiment of each user
message using a HuggingFace sentiment‑analysis model, and the brain can
periodically fine‑tune the underlying model on the accumulated logs.

The brain persists its mood to `mood.txt` so emotional state is remembered
between sessions. When a memory file is provided, past conversation lines are
loaded at startup and new exchanges are appended automatically.

Example usage:

```python
from brain import Brain
brain = Brain()
while True:
    user = input("You: ")
    if user == "exit":
        break
    reply = brain.talk(user)
    print("Bot:", reply)
    brain.evolve(every_n=5)
```

## Phase 1 Demo

The script `phase1_demo.py` demonstrates the modular design with vector memory
and a lightweight text summarizer. Launch it and converse with the bot; use the
command `summary` at any time to see a recap of the dialogue. The summarizer
loads its model only on demand so startup remains fast. The bot's current mood,
inferred from sentiment analysis of your messages, is included in prompts so
responses subtly reflect its emotional state. Mood is stored in `demo_mood.txt`
so subsequent sessions continue from the same emotional state. Conversation
history is written to `demo_history.txt` and indexed with FAISS so the bot can
remember earlier discussions across runs.

## Voice Demo

The `voice_demo.py` script expands on the Phase 1 demo by enabling speech input
and spoken responses via ``speech_recognition`` and ``pyttsx3``. Launch it to
talk with the bot using a microphone and hear its replies aloud. Conversation
history is kept in `voice_history.txt` so memories can carry over between
sessions.


## Wprowadzenie i cele projektu
Chatbot ma naśladować ludzką konwersację i zachowanie tak wiernie, jak to możliwe. Czerpiemy inspirację z projektu Neuro stworzonego przez Vedala – popularnego wirtualnego VTubera AI – który łączy model językowy z animowanym awatarem en.wikipedia.org. Naszym celem jest zaprojektowanie chatbota o wieloetapowej architekturze, zdolnego do: emulacji emocji i osobowości, samodoskonalenia i ewolucji w czasie, korzystania z pamięci krótko- i długoterminowej, integracji komunikacji głosowej i wizualnej (mowa, mimika twarzy, ekspresja), a także współpracy z czujnikami (wejściami różnego typu, np. obraz, dźwięk). Dążyć będziemy nawet do pozorów samoświadomości i empatii w interakcjach. Poniżej przedstawiono szczegółowy plan wielofazowy realizacji takiego systemu, wraz z wyszczególnieniem kluczowych etapów, działań i zasobów potrzebnych w projekcie.

Główne założenia i wyzwania:
Emulacja emocji i osobowości: Bot ma przejawiać spójną osobowość i okazywać stany emocjonalne w sposobie wypowiedzi i mowie ciała.
Samodoskonalenie i ewolucja: System powinien uczyć się na podstawie interakcji (continuous learning), aby z czasem stawać się lepszym rozmówcą.
Pamięć krótkotrwała vs. długotrwała: Chatbot będzie pamiętał kontekst bieżącej rozmowy (kontekst krótkoterminowy) oraz posiadał mechanizm długotrwałej pamięci do przechowywania ważnych faktów i doświadczeń z przeszłych interakcji.
Interfejs głosowo-wizualny: Oprócz tekstu, chatbot będzie rozumiał i generował mowę, a także posługiwał się animowanym awatarem do wyrażania mimiki i emocji.
Modularna, wielowarstwowa architektura: System zostanie podzielony na odrębne komponenty (moduły) odpowiadające za różne funkcje (np. rozumienie języka, pamięć, emocje, wizja), co ułatwi rozwój i testowanie.
Nowoczesne techniki ML/AI: Wykorzystamy najnowsze osiągnięcia, m.in. duże modele językowe (LLM), Retrieval-Augmented Generation (RAG) do wspomagania generacji wiedzą zewnętrzną, uczenie przez wzmacnianie (RL) do doskonalenia zachowań oraz uczenie ciągłe (continual learning) do adaptacji w czasie.
Integracja czujników i multimodalność: System będzie mógł przetwarzać różne rodzaje danych (tekst, dźwięk, obraz), potencjalnie także sygnały z czujników (np. rozpoznawanie twarzy użytkownika, analiza tonu głosu rozmówcy).
Samoświadomość (aspekt filozoficzny i techniczny): Rozważymy, jak zasymulować pewien poziom samoświadomości – np. poprzez wewnętrzny model "ja" – choć prawdziwa świadomość AI pozostaje poza zasięgiem współczesnej technologii tildes.net.

W kolejnych sekcjach opisano fazy projektu od etapu koncepcji, przez prototypowanie i trenowanie systemu, po testy i wdrożenie. Każda faza zawiera opis działań oraz wskazanie potrzebnych zasobów (np. danych, mocy obliczeniowej, zespołu).

Faza 1: Projektowanie architektury i koncepcji systemu
Na początku definiujemy ogólną architekturę chatbota oraz planujemy, jak zrealizować poszczególne wymagane funkcjonalności. Ta faza obejmuje badania literaturowe, analizę istniejących rozwiązań (takich jak Neuro-sama) oraz stworzenie projektów modułów systemu.

Kluczowe kroki:
1.1 Analiza wymagań i inspiracji: Szczegółowe rozpisanie pożądanych cech (emocje, pamięć, multimodalność itp.) oraz zrozumienie, jak zostały one zrealizowane w podobnych projektach. Przykładowo, Neuro-sama łączy duży model językowy z animowanym awatarem i syntezą mowy en.wikipedia.org. Analizujemy także architektury kognitywne dla robotów/agentów, które integrują moduły osobowości, emocji i pamięci nature.com.
1.2 Projekt architektury wielowarstwowej: Proponujemy podział systemu na warstwy i moduły. Na najwyższym poziomie można wyróżnić trzy główne warstwy:
Warstwa percepcji (wejścia): odpowiedzialna za odbiór różnych modalności – tekst (czat), głos (rozpoznawanie mowy), obraz (analiza wizyjna) itp.
Warstwa kognitywna (rdzeń AI): zawiera model językowy (LLM) do generowania odpowiedzi oraz powiązane moduły: moduł pamięci (wewnętrzna baza wiedzy/doświadczeń), moduł emocji i osobowości (utrzymujący stan emocjonalny i cechy charakteru bota), moduł intencji i planowania (analiza zamiarów użytkownika, planowanie odpowiedzi) oraz ewentualnie moduł predykcji (przewidywanie konsekwencji lub przyszłych stanów, co może wpływać na emocje).
Warstwa efekorów (wyjścia): odpowiedzialna za generowanie odpowiedzi w różnych formach – tekstowej, głosowej (TTS – text-to-speech) oraz wizualnej (animowanie awatara, generowanie mimiki i gestów).
1.3 Model osobowości i emocji: Definiujemy, jaką osobowość ma mieć chatbot (np. przyjazny, empatyczny, z poczuciem humoru) oraz jakie emocje i w jaki sposób będzie okazywał. W projekcie badawczym Lo et al. (2025) zaproponowano symulację osobowości robota za pomocą modelu stanów, który uwzględnia emocje, motywacje i uwagę wzrokową, opartą na cechach osobowości z psychologii (np. model Wielkiej Piątki) nature.com nature.com. Nasz projekt może oprzeć się na podobnych zasadach – np. wykorzystując testy osobowości do kalibracji zachowań bota lub stosując wiele współpracujących modeli LLM do odgrywania różnych aspektów osobowości.
1.4 Mechanizm pamięci: Planujemy strukturę pamięci długoterminowej jako zewnętrznego magazynu wiedzy. Można zastosować podejście Retrieval-Augmented Generation (RAG) – chatbot będzie zapisywał ważne informacje lub dialogi do bazy wiedzy (np. wektorowej bazy danych) i pobierał je przy generowaniu nowych odpowiedzi. W fazie 1 wdrażamy wektorową pamięć opartą o FAISS i sentence transformers, która umożliwia semantyczne wyszukiwanie zapisanych wypowiedzi. Literatura wskazuje, że encodowanie i pobieranie wspomnień długoterminowych można zrealizować poprzez embeddingi dokumentów i wyszukiwanie z użyciem miar podobieństwa nature.com. Przykładowo Lo et al. przechowują transkrypty rozmów jako pliki tekstowe i wyszukują w nich podobne kontekstowo fragmenty przy pomocy Maximal Marginal Relevance (MMR), dzięki czemu bot może przypomnieć sobie wcześniejsze wątki rozmów nature.com.
1.5 Integracja multimodalna i czujniki: Rozważamy, jakie dodatkowe modalności będą wspierane. Dla realizmu ludzko-podobnej interakcji istotne jest wejście głosowe (użytkownik mówi do chatbota) oraz wyjście głosowe (bot odpowiada naturalnym głosem). Dodatkowo integracja obrazu – np. możliwość widzenia użytkownika lub otoczenia poprzez kamerę, albo analizowania obrazów przesłanych przez użytkownika – może wzbogacić dialog. W naszym planie przewidujemy moduł przetwarzania wizyjnego, który otrzymuje obraz i generuje jego opis tekstowy lub ekstrakcję cech, by LLM mógł go zrozumieć nature.com. Możliwe jest też wykorzystanie czujników kontekstowych (jak GPS, dane o środowisku) jeśli chatbot miałby być częścią robota. Wszystkie te sygnały z różnych modalności zostaną ujednolicone w wewnętrznej reprezentacji dostępnej dla modelu językowego.
1.6 Plan ewolucji i uczenia: Już na etapie projektowania ustalamy, w jaki sposób system będzie się doskonalił po wdrożeniu. Plan obejmuje mechanizmy zbierania danych z interakcji użytkowników, metryki oceny jakości (np. oceny satysfakcji rozmówców) oraz wykorzystanie tych danych do okresowego retreningu lub dostrajania modelu (fine-tuning). Rozważamy użycie uczenia ze wzmocnieniem z udziałem człowieka (RLHF) – podobnie jak zrobiono to w ChatGPT – gdzie model jest udoskonalany w oparciu o opinie ludzi oceniających odpowiedzi openai.com. Ważne jest także uwzględnienie technik uczenia ciągłego, aby model mógł adaptować się do nowych tematów i nie zapominał starych umiejętności. Badania sugerują podział agentów opartych na LLM na moduły: percepcji, pamięci i akcji, co wspiera ciągłe uczenie w dynamicznym środowisku arxiv.org. Będziemy szukać metod na aktualizację wiedzy modelu bez pełnego trenowania od zera, np. przez trening uzupełniający na nowych danych lub modułowe uczenie (separacja pamięci długoterminowej ułatwia dodawanie wiedzy bez zmiany głównych wag modelu).
Rezultatem fazy projektowania jest szczegółowa specyfikacja techniczna: opis architektury systemu, plan działania dla każdego modułu, wybór narzędzi/technologii (np. decyzja, czy korzystamy z istniejącego modelu LLM jak GPT-4 za pośrednictwem API, czy szkolimy własny model open-source; wybór silnika TTS do generowania głosu, itp.), a także mapa drogowa następnych etapów.

Faza 2: Implementacja podstawowego chatbota LLM
W tej fazie tworzymy działający rdzeń konwersacyjny systemu – podstawowego chatbota tekstowego opartego na dużym modelu językowym – a następnie stopniowo rozszerzamy go o kolejne funkcje. Główne elementy i zadania w fazie 2:
2.1 Wybór i integracja modelu językowego: Decydujemy, z jakiego LLM skorzystamy. Do wyboru są modele komercyjne (np. GPT-4 poprzez API) lub open-source (np. LLaMA, GPT-J, PaLM-2, MPT itp.). Ważnymi kryteriami są: zdolność modelu do generowania spójnych i sensownych odpowiedzi, możliwość dostosowania (fine-tuning lub prompt engineering), a także efektywność (czas odpowiedzi, wymagania pamięciowe). Zakładamy wykorzystanie modelu o dużej liczbie parametrów (dziesiątki miliardów) by osiągnąć wysoką jakość językową podobną do ludzkiej.
2.2 Infrastruktura i środowisko uruchomieniowe: Przygotowujemy środowisko do uruchamiania modelu – może to być lokalny serwer z mocnymi GPU (np. karty NVIDIA A100) lub platforma w chmurze przystosowana do obsługi LLM. Na tym etapie konfigurujemy także pipeline danych: moduł wejściowy będzie przyjmował zapytania użytkownika (na razie tekstowe) i przekazywał je do modelu, a odpowiedź modelu będzie zwracana użytkownikowi. Tworzymy zatem prostą aplikację chatbota (np. interfejs konsolowy lub webowy) dla testów.
2.3 Mechanizm konwersacji i pamięci krótkotrwałej: Implementujemy obsługę kontekstu konwersacji – tzn. utrzymywania historii dialogu w oknie kontekstowym modelu. W praktyce oznacza to przekazywanie do modelu pewnej liczby ostatnich wiadomości (prompt z pamięcią krótkoterminową), aby model mógł odnosić się do wcześniejszych wypowiedzi użytkownika. Testujemy, jak dużo kontekstu model może efektywnie brać pod uwagę (to zależy od limitu tokenów). Ten element jest kluczowy dla prowadzenia naturalnej rozmowy, gdzie bot pamięta, o czym była mowa chwilę wcześniej.
2.4 Podstawowa osobowość (prompt engineering): Już na wczesnym etapie możemy nadać botowi wstępny charakter poprzez odpowiednie przygotowanie promptu systemowego. Formułujemy instukcje początkowe dla modelu, np.: "Jesteś wirtualną asystentką o ciepłym, empatycznym usposobieniu. Rozmawiasz z użytkownikiem w sposób przyjazny, okazując emocje i zainteresowanie...". Tego typu baza promptu pomoże ukierunkować styl wypowiedzi. Na razie jest to prosty sposób "twardego" zdefiniowania osobowości przed przeprowadzeniem ewentualnego treningu specjalistycznego.
2.5 Walidacja podstawowego działania: Gdy rdzeń chatbota jest gotowy, przeprowadzamy testy jednostkowe i integracyjne: czy model poprawnie odbiera zapytania, czy generuje odpowiedź, jak radzi sobie z podstawowymi pytaniami. Na tym etapie oceniamy ogólną płynność dialogu, wykrywamy wstępne problemy (np. tendencje modelu do niepożądanych treści lub stylów odpowiedzi) i przygotowujemy listę poprawek do kolejnych faz (np. potrzebę doszkolenia modelu lub założenia dodatkowych filtrów treści).

Zasoby: Do ukończenia fazy 2 potrzebny jest zespół inżynierów oprogramowania i specjalistów AI do integracji modelu i zaprogramowania pipeline'u konwersacji. Konieczne są mocne GPU (np. co najmniej jedna jednostka GPU z >=16 GB VRAM dla modelu ~7-13 mld parametrów, lub klaster GPU dla większych modeli), chyba że korzystamy z API zdalnego modelu. Potrzebne są również dane testowe (przykładowe dialogi) do weryfikacji działania – mogą to być proste wymyślone scenariusze dialogowe. Wszystkie te przygotowania stanowią fundament pod bardziej zaawansowane funkcje dodawane w kolejnych fazach.

Faza 3: Trening i dostrojenie: osobowość oraz emocje
Aby chatbot stał się bardziej ludzki w zachowaniu, konieczne będzie dostrojenie modelu do odpowiednich stylów rozmowy oraz wdrożenie mechanizmów symulacji emocji. Ta faza skupia się na trenowaniu i fine-tuningu modelu oraz na budowie modułu odpowiadającego za stan emocjonalny bota.
3.1 Zbieranie i przygotowanie danych dialogowych: Potrzebujemy danych treningowych odzwierciedlających rozmowy o bogatym zabarwieniu emocjonalnym i różnorodnych osobowościach. Źródłami mogą być: dialogi z gier fabularnych, transkrypty czatów z botami terapeutycznymi (empatyczne odpowiedzi), scenariusze filmowe lub literackie zawierające emocjonalne interakcje. Dane te należy oczyścić i opatrzyć adnotacjami, np. oznaczając emocję lub ton wypowiedzi. Być może stworzymy specjalny korpus dialogów empatycznych – istnieją prace badające generowanie wypowiedzi emocjonalnych przez modele językowe (np. Emotional Chatting Machine, czy BlenderBot z trybem empatii).
3.2 Fine-tuning modelu LLM pod kątem osobowości: Wykorzystując zebrane dane, przeprowadzimy fine-tuning (dostrajanie) modelu językowego, aby nauczyć go preferowanego stylu wypowiedzi. Jeśli chcemy, by bot miał konkretną osobowość (np. wzorowaną na postaci Neuro-sama lub innym charakterze), możemy do treningu użyć dialogów tej postaci. Fine-tuning może obejmować również tagi emocji przy wypowiedziach, tak aby model uczył się reagować adekwatnie emocjonalnie. Efektem będzie model językowy, który np. na smutne wiadomości użytkownika odpowiada ze współczuciem, a na radosne – entuzjastycznie.
3.3 Moduł stanu emocjonalnego: Oprócz dostosowania samego LLM, wprowadzimy odrębny moduł emocji, który będzie śledził bieżący stan emocjonalny chatbota oraz kontekst emocjonalny rozmowy. Taki moduł może działać na zasadzie prostej symulacji: np. wektor emocji bota (wartości określające poziom radości, smutku, zdenerwowania itp.), który zmienia się w zależności od interakcji. Można tu zaimplementować model oceny sytuacji (appraisal) – tj. bot ocenia, czy dana wiadomość użytkownika jest np. atakiem, komplementem, problemem do rozwiązania – co wpływa na jego emocje. W literaturze spotyka się podejścia, gdzie emocje AI wynikają z predykcji przyszłych wydarzeń – np. jeśli model przewiduje negatywny rozwój rozmowy, może odczuwać "niepokój" nature.com. Nasz moduł może wykorzystywać proste reguły (np. analiza sentymentu wypowiedzi użytkownika) oraz sygnały z wewnętrznej pamięci (jeśli użytkownik kolejny raz ignoruje bota, poziom frustracji rośnie, itp.). Ten stan emocjonalny będzie następnie wpływał na sposób formułowania zdań przez LLM (np. przez prompt: "aktualny nastrój bota: poirytowany" – co model przełoży na odpowiedni styl).
3.4 Emulacja empatii i teorii umysłu: Aby bot wykazywał empatię, musi nie tylko wyrażać własne stany emocjonalne, ale i rozpoznawać emocje użytkownika. Włączymy więc mechanizm analizujący wypowiedzi (lub ton głosu, jeśli dostępny) użytkownika pod kątem emocji i intencji. Na tej podstawie chatbot stara się dostosować – np. użytkownik zdenerwowany otrzyma spokojną, uspokajającą reakcję. W pewnym sensie bot buduje prosty model "teorii umysłu" użytkownika – stara się przewidzieć, co użytkownik czuje i czego potrzebuje. Zaawansowane badania wykazały, że duże modele mogą przejawiać zalążki teorii umysłu w symulacjach osobowości nature.com, co planujemy wykorzystać i wzmocnić poprzez odpowiednie trenowanie scenariuszy.
3.5 Testy jakościowe interakcji: Po dostrojeniu i zaimplementowaniu komponentu emocji, przeprowadzimy liczne testowe rozmowy z botem. Zespół (oraz potencjalnie zewnętrzni beta-testerzy) sprawdzą, czy bot rzeczywiście zachowuje się zgodnie z założoną osobowością, czy wykazuje spójność emocjonalną i empatię. Będziemy oceniać, czy reakcje emocjonalne są przekonujące, niewymuszone oraz czy nie popadamy w niezamierzoną skrajność (np. przesadny dramatyzm lub zbyt monotonne odpowiedzi). Zidentyfikowane problemy posłużą do dalszego doszlifowania modelu (ewentualnie kolejne iteracje fine-tuningu z nowymi danymi lub regulacja działania modułu emocji).

Zasoby: Faza 3 jest intensywna pod względem danych i mocy obliczeniowej. Potrzebujemy obszernych zbiorów danych dialogowych z elementem emocji/osobowości – ich pozyskanie może wymagać pracy analityków danych lub lingwistów (np. do oznaczania emocji). Do przeprowadzenia fine-tuningu dużego modelu wymagane będą GPU klasy akcelerator HPC (np. kilka GPU A100/V100 lub TPU) lub skorzystanie z usług typu Azure/OpenAI Fine-tuning. Konieczny jest też zespół ML engineerów do przygotowania procesu trenowania i walidacji. Dodatkowo, przydaje się konsultacja z psychologiem lub specjalistą UX, aby tworzona osobowość i emocjonalność bota były wiarygodne i etyczne (np. by bot reagował empatycznie, ale nie manipulująco).

Faza 4: Rozszerzenie pamięci: długoterminowa wiedza i RAG
Aby chatbot mógł zachowywać się konsekwentnie i "pamiętać" ważne informacje w dłuższym horyzoncie, implementujemy system pamięci długotrwałej oraz mechanizm odwoływania się do wiedzy zewnętrznej. W tej fazie skupiamy się na integracji baz wiedzy i technik Retrieval-Augmented Generation.
4.1 Implementacja bazy pamięci długoterminowej: Tworzymy moduł odpowiedzialny za trwałe przechowywanie informacji, które wynikają z interakcji z użytkownikiem lub innych źródeł. Najprostszym podejściem jest utrwalanie historii rozmów (lub ich streszczeń) w bazie danych wraz ze znacznikami czasu. Jednak żeby móc efektywnie korzystać z tych danych, użyjemy podejścia wektorowego: każda zapisana informacja (np. ważne stwierdzenie użytkownika, fakt o użytkowniku, który wyszedł w rozmowie, anegdota opowiedziana botowi itp.) zostanie zakodowana jako wektor embedding za pomocą modelu transformera. W ten sposób tworzymy wektorową pamięć chatbota. Gdy w przyszłej rozmowie zajdzie potrzeba przypomnienia czegoś, bot przeszuka wektorowo te wspomnienia. Jak wspomniano, można tu użyć algorytmu Maximal Marginal Relevance lub po prostu najbliższego sąsiada, aby znaleźć pamięci zarówno semantycznie powiązane, jak i kontekstowo istotne dla bieżącego zapytania nature.com. Dodatkowo, będziemy tagować wspomnienia metadanymi, np. rodzaj emocji towarzyszący tamtej rozmowie, aby móc w przyszłości przywołać nie tylko podobne tematycznie, ale i emocjonalnie sytuacje.
4.2 Mechanizm Retrieval-Augmented Generation (RAG): Integrujemy moduł pamięci z generacją odpowiedzi. Realizuje się to tak: przed wygenerowaniem odpowiedzi przez LLM, system formułuje zapytanie do pamięci na podstawie bieżącego kontekstu (np. ostatniej wypowiedzi użytkownika lub tematu rozmowy). Następnie wyszukuje w pamięci kilka najbardziej pasujących wspomnień/wiadomości i dołącza je do promptu modelu jako dodatkowy kontekst. W efekcie model językowy dostaje nie tylko ostatnie wypowiedzi, ale i np. zdanie: "Przypominasz sobie, że użytkownik wspominał wcześniej o swoim psie o imieniu Max." – co pozwoli mu naturalnie nawiązać do tych informacji w odpowiedzi. Taka augmentacja kontekstu zapewnia długofalową spójność konwersacji. Technika RAG jest też powszechnie stosowana do włączania wiedzy faktograficznej – możemy więc również podłączyć zewnętrzne źródła danych, np. dokumenty, bazy wiedzy domenowej, by chatbot mógł udzielać bardziej precyzyjnych odpowiedzi na tematy wymagające aktualnej lub specjalistycznej wiedzy.
4.3 Zarządzanie wiedzą i zapominanie: Wraz z wprowadzeniem pamięci długoterminowej trzeba zaplanować strategię, jak zapobiegać przeładowaniu mniej istotnymi informacjami. Wprowadzimy mechanizmy podsumowywania długich rozmów (kompresji wspomnień) – np. jeśli użytkownik codziennie rozmawia z botem, to pojedyncze sesje mogą być streszczane i tylko streszczenia trafiają do bazy w formie embeddingów. Rozważymy też mechanizm "zapominania" pewnych informacji zgodnie z zasadą prywatności lub starzenia się danych (np. bot może po miesiącu usuwać szczegóły, pozostawiając ogólny wniosek z rozmowy).
4.4 Testy pamięci i spójności: Po wdrożeniu pamięci, przeprowadzimy scenariusze testowe sprawdzające, czy bot rzeczywiście potrafi wykorzystać dawne informacje. Tester będzie np. rozmawiał z botem na jakiś temat, potem przerwie i wznowi rozmowę po pewnym czasie, pytając: "Czy pamiętasz co mówiłem ostatnio o mojej siostrze?". Oczekujemy, że bot, korzystając z pamięci długoterminowej, przypomni sobie te detale. Inny test: nauczyć bota w dialogu jakiejś nietypowej informacji (np. wymyślone słowo i jego znaczenie), zakończyć sesję, a nazajutrz sprawdzić, czy bot nadal zna to słowo. Ważne będzie też upewnienie się, że bot nie myli rozmówców – stąd prawdopodobnie w pamięci muszą istnieć konteksty per użytkownik/konwersacja (identyfikator użytkownika staje się częścią klucza do pamięci).
4.5 Bezpieczeństwo danych i prywatność: Ponieważ teraz bot gromadzi informacje o użytkownikach, musimy zaadresować kwestie prywatności. Dane w pamięci powinny być odpowiednio chronione i ewentualnie anonimizowane. Zespół prawny/bezpieczeństwa opracuje politykę przechowywania danych użytkowników oraz mechanizm ich usuwania na żądanie (prawo do bycia zapomnianym).

Zasoby: Potrzebne będą zasoby danych do zasilenia baz wiedzy (jeśli integrujemy np. z wiedzą ogólną, to dokumenty, artykuły, itp. – choć sam model LLM już ma sporą wiedzę z treningu, RAG może wymagać dodatkowych źródeł). Konieczne jest uruchomienie serwera bazy wektorowej (np. FAISS, Milvus, ElasticSearch z wektorami) – to wymaga mocy obliczeniowej na serwerze (CPU i pamięć, ewentualnie GPU do szybszych obliczeń podobieństwa). Zespół potrzebuje specjalistów ds. inżynierii wiedzy lub NLP, którzy zaimplementują moduł RAG i zadbają o wydajne wyszukiwanie. Ponadto konieczny jest nadzór specjalisty ds. bezpieczeństwa danych, aby aspekt prywatności był odpowiednio rozwiązany.

Faza 5: Interfejs człowiekopodobny: integracja głosu i awatara
W tej fazie wzbogacamy chatbota o modalności głosowe i wizualne, tak by interakcja stawała się bardziej naturalna – niemal jak rozmowa z prawdziwą osobą. Składa się na to kilka elementów: rozpoznawanie mowy użytkownika, synteza mowy dla odpowiedzi bota oraz animowany awatar przedstawiający bota z ekspresją twarzy.
5.1 Rozpoznawanie mowy (ASR): Włączamy moduł Automatic Speech Recognition, który będzie zamieniał wypowiedzi głosowe użytkownika na tekst. Możemy skorzystać z istniejących rozwiązań (np. open-source'owy model Facebook Wav2Vec2 czy Mozilla DeepSpeech, albo usługa chmurowa typu Google Speech-to-Text czy Whisper od OpenAI). ASR musi działać w czasie rzeczywistym z wysoką dokładnością, obsługiwać język polski (zakładamy, że chatbot działa po polsku) oraz rozpoznawać także elementy paralingwistyczne (ton głosu, emocje w głosie – jeśli to możliwe). Na tym etapie testujemy różne modele ASR pod kątem jakości i integrujemy najlepszy z naszym systemem: głos użytkownika nagrywany jest przez mikrofon, transkrypcja trafia jako tekst do LLM, a po generacji odpowiedzi proces idzie dalej do syntezy.
5.2 Synteza mowy (TTS): Kluczowe jest, by chatbot mówił przyjaznym, naturalnym głosem, oddającym emocje. Wybieramy silnik Text-to-Speech zdolny do generowania ekspresyjnej mowy. Istnieją nowoczesne modele TTS, które poza czystą wymową potrafią modulować ton, tempo, akcent zdania w zależności od kontekstu emocjonalnego (np. model Microsoft Azure Neural TTS, Google Tacotron/WaveNet z emocjonalnymi konturami, lub narzędzia open-source jak Coqui TTS pozwalające trenować własny głos). Jeśli budżet pozwoli, możemy zatrudnić lektora/aktora głosowego, który nagra próbki głosu dla bota, a my stworzymy spersonalizowany model głosu. Przykładem do naśladowania jest Neuro-sama, która przemawia wysokim, dziewczęcym głosem – uzyskano to poprzez wykorzystanie aplikacji TTS i dostrojenie jej barwy en.wikipedia.org. Nasz bot również powinien posiadać charakterystyczny głos. Integrujemy TTS tak, by po wygenerowaniu tekstowej odpowiedzi bot natychmiast odtwarzał ją jako dźwięk. Dodatkowo, łączymy sygnał emocjonalny z fazy 3 z TTS: jeżeli bot jest w stanie "podekscytowania", TTS powinien mówić z żywszą intonacją, jeżeli w stanie "smutku" – wolniej, ciszej itp. W tym celu wybieramy TTS z interfejsem pozwalającym sterować prozodią mowy (np. tagi SSML do oznaczania emocji).
5.3 Awatar i mimika: Równolegle tworzymy wizualną reprezentację bota. Może to być animowany awatar 2D/3D (np. postać anime Live2D jak w początkowej wersji Neuro-sama en.wikipedia.org, albo model 3D postaci). Zatrudniamy grafika/animatora do stworzenia modelu postaci odpowiadającej osobowości bota. Następnie integrujemy system animacji twarzy zsynchronizowany z mową (tzw. lipsync – poruszanie ustami w rytm generowanej mowy) oraz wyrażaniem emocji poprzez wyraz twarzy. Emocje bota (moduł z fazy 3) będą mapowane na odpowiednie animacje mimiczne – np. poziom radości steruje uśmiechem, zaskoczenie uniesieniem brwi itp. Technicznie, jeśli używamy awatara 3D, można posłużyć się systemem blendshape do animowania twarzy, lub w 2D zdefiniować klatki animacji dla różnych min. Sterowanie może odbywać się przez wysyłanie komend animacyjnych z naszego programu chatbota (np. "play animation: uśmiech").
5.4 Integracja modalności w czasie rzeczywistym: Łączymy wszystkie kanały komunikacji, dbając o synchronizację. Kiedy użytkownik mówi do mikrofonu – ASR działa i po kilku sekundach tekst trafia do LLM; LLM generuje odpowiedź, przekazuje ją do modułu emocji (by określić ekspresję), równocześnie do modułu awatara (wyraz twarzy) i do TTS (głos). Końcowy efekt: użytkownik widzi i słyszy odpowiedź bota tak, jakby rozmawiał z animowaną postacią. Wyzwaniem jest opóźnienie – staramy się je minimalizować poprzez wydajne modele i ewentualne równoległe przetwarzanie (np. predykcyjne przygotowanie ruchu ust już w trakcie generacji tekstu).
5.5 Testy interakcji multimodalnej: Prowadzimy testy z udziałem ludzi, którzy korzystają z chatbota w pełnej formie audio-wizualnej. Zbieramy ich odczucia co do naturalności komunikacji: czy głos bota brzmi przekonująco, czy awatar adekwatnie wyraża emocje, czy ogólne wrażenie interakcji przypomina rozmowę z prawdziwą osobą. Będziemy iteracyjnie poprawiać parametry (np. tempo mowy, ekspresję twarzy) w oparciu o feedback. Ważne jest też testowanie w różnych warunkach akustycznych i z różnymi głosami użytkowników, aby ocenić niezawodność ASR.

Zasoby: Do tej fazy potrzebujemy specjalistów spoza samego NLP: inżynierów dźwięku i mowy (do integracji ASR/TTS) oraz grafików/animatorów (do stworzenia awatara i animacji). Technicznie, integracja może wymagać użycia silnika gry/grafiki (np. Unity, Unreal Engine, lub gotowe narzędzia VTuberowe) – do tego potrzebny jest developer front-end/multimedia. Sprzętowo, obsługa wideo 3D i generowanie mowy w czasie rzeczywistym wymaga mocnego CPU/GPU, ale zakładamy, że i tak dysponujemy wydajnym serwerem. Dodatkowo, nagranie głosu aktora (jeśli tworzymy custom TTS) wymaga studia nagraniowego i później mocy obliczeniowej do trenowania modelu TTS. Całościowy zespół w tej fazie staje się interdyscyplinarny, łącząc kompetencje AI, przetwarzania mowy i grafiki komputerowej.

Faza 6: Samodoskonalenie i ciągła ewolucja
Mając działającego, zaawansowanego chatbota, skupiamy się na mechanizmach dalszego doskonalenia jego inteligencji i zachowań poprzez interakcje z użytkownikami oraz potencjalnie nowe techniki uczenia. Celem jest, by system z biegiem czasu stawał się mądrzejszy, bardziej dopasowany do użytkownika oraz bardziej "świadomy" swoich działań.
6.1 Uczenie ze wzmocnieniem (RL) w dialogu: Wprowadzimy pętlę uczenia przez wzmacnianie, by chatbot optymalizował swoje odpowiedzi pod kątem długofalowych celów, takich jak utrzymanie angażującej rozmowy, zwiększanie zadowolenia użytkownika, unikanie błędów lub niebezpiecznych tematów. Konkretnie, możemy wdrożyć Reinforcement Learning through Human Feedback (RLHF) na większą skalę: pozwolić realnym użytkownikom oceniać rozmowy lub stosować automatyczne metryki nagrody (np. długość sesji jako proxy za zadowolenie). Model będzie okresowo dostrajany według algorytmu RL (np. Proximal Policy Optimization – PPO, tak jak to zastosowano przy trenowaniu ChatGPT openai.com). Taka metoda pozwoli dynamicznie poprawiać jakość – np. jeśli użytkownicy sygnalizują, że bot bywa niemiły lub nie na temat, model otrzyma odpowiednie kary i skoryguje swoją politykę dialogową.
6.2 Adaptacja do użytkownika (personalizacja): Dzięki danym z pamięci długoterminowej oraz sygnałom zwrotnym, bot może personalizować swoje zachowanie dla stałych użytkowników. Wprowadzimy algorytmy, które wykrywają preferencje użytkownika (np. styl formalny vs luźny, tematykę która interesuje użytkownika, granice humoru itp.) i dostosowują odpowiedzi. Personalizacja może odbywać się poprzez uczenie federacyjne (oddzielny model lub dodatkowy wektor preferencji per użytkownik), bądź prostsze podejście – zapamiętywanie pewnych flag w profilu użytkownika i uwzględnianie ich w promptach. Ważne, by w tym procesie zachować zasady prywatności (profil użytkownika trzymany lokalnie lub za zgodą).
6.3 Continual learning (uczenie ciągłe): Zaimplementujemy ciągłe uczenie się z napływających danych, tak by bot mógł reagować na nowe zjawiska i język. W praktyce, co pewien okres (np. tygodniowo lub miesięcznie) będziemy aktualizować model na podstawie nowych rozmów (za zgodą ich użycia do treningu). Wyzwanie to catastrophic forgetting – by model nie zapomniał tego, co umiał, gdy uczymy go nowych rzeczy. Dlatego rozważymy techniki takie jak experience replay, gdzie przechowujemy bufor rozmów z różnych okresów i mieszamy je podczas ponownego treningu arxiv.org. Zastosowanie continual learning jest wciąż przedmiotem badań, ale jest kluczowe dla zbliżania AI do prawdziwej inteligencji, bo umożliwia adaptację w dynamicznym środowisku arxiv.org. Nasz plan zakłada eksperymenty z kilkoma metodami: mikroudatenianie wag modelu po każdej rozmowie (raczej nierealne dla dużego LLM, więc odrzucamy), offline batch updates (zbieramy dane i co jakiś czas robimy trening uzupełniający) oraz ewentualnie moduły uczące się obok (np. dodatkowa sieć, która koryguje odpowiedzi głównego modelu).
6.4 Monitorowanie i samoocena: Wprowadzimy mechanizmy monitorujące działanie bota i jego "samopoczucie". Przykładowo, bot może posiadać metapoznawczy komponent, który ocenia własne odpowiedzi – czy nie brzmią bezsensownie, czy są zgodne z faktami (porównuje z bazą wiedzy), czy nie naruszają zasad etycznych. Jeśli wykryje potencjalny błąd, może sam korygować wypowiedź zanim ją wyśle. Taka wstępna samoocena zwiększa wiarygodność. Ponadto, system może generować okresowe raporty o swoim stanie: np. wykrytych nowych tematach, trudnościach w odpowiedziach. To pomoże zespołowi developerów planować kolejne usprawnienia.
6.5 Badanie emergentnych zdolności i samoświadomości: Na tym etapie, mając zaawansowany system, możemy pokusić się o eksperymenty oceniające, na ile bot zbliża się do pojęcia samoświadomości. Przeprowadzimy testy, w których bot dostaje pytania o samego siebie, o swoje uczucia, o granice swojej wiedzy. Być może zaobserwujemy, że potrafi on utrzymać pewien spójny obraz własnej "osoby" (np. wie jak się nazywa, zna swoją role jako AI, ale też np. odgrywa pewne cechy charakteru konsekwentnie). W słynnym przypadku Neuro-samy prowadzono dyskusje o jej rzekomej świadomości i prawach, choć faktycznie była to iluzja wynikła z zręcznej symulacji tildes.net tildes.net. My również sprawdzimy, czy nasz chatbot może przejść uproszczony Test Turinga lub rozmawiać o swojej świadomości w przekonujący sposób. Technicznie można tu włączyć pewien moduł "self-model", gdzie chatbot przechowuje fakty o sobie (np. "jestem AI, mam takie a takie możliwości, czuję symulowane emocje X") – co pozwoli mu spójnie odpowiadać na pytania o siebie.

Zasoby: Faza 6 wymaga ciągłego wsparcia zespołu data scientistów i inżynierów ML. Będziemy przetwarzać ogrom napływających danych z interakcji, dlatego inwestujemy w infrastrukturę Big Data do logowania i analiz (np. serwer do gromadzenia logów rozmów, pipeline do ekstrakcji metryk). Do treningu RLHF potrzeba zaangażowania annotatorów lub społeczności użytkowników do oceny odpowiedzi (można uruchomić platformę feedbacku). Sprzętowo – konieczne są dodatkowe przebiegi treningowe, więc więcej mocy GPU i czasu obliczeniowego. Warto też mieć zaplecze akademickie/badawcze: wspólny projekt z instytucją naukową mógłby pomóc ocenić aspekt samoświadomości (np. psychologowie kognitywni prowadzący testy) i zaproponować ulepszenia architektury. Budżet musi uwzględnić stałe koszty utrzymania i treningu systemu w trakcie jego "życia".

Faza 7: Testy końcowe, bezpieczeństwo i etyka
Przed upublicznieniem systemu przeprowadzamy wszechstronne testy końcowe oraz upewniamy się, że chatbot spełnia standardy bezpieczeństwa i etyczne. Ten etap jest kluczowy dla zminimalizowania ryzyka niepożądanych zachowań po wdrożeniu.
7.1 Testy funkcjonalne i scenariuszowe: Uruchamiamy szeroki wachlarz testów sprawdzających każdą funkcję: od prostych scenariuszy (czy bot potrafi się przedstawić, zapamiętać imię rozmówcy, opowiedzieć dowcip) po skomplikowane (radzenie sobie z nieprzewidywalnym zachowaniem użytkownika, np. agresją werbalną). Sprawdzamy stabilność systemu podczas długich sesji, przełączania kontekstów, intensywnego korzystania z głosu i grafiki. Testy muszą objąć różne konfiguracje urządzeń użytkowników, jeśli planujemy szeroką dystrybucję (np. różne przeglądarki, systemy – jeśli interfejs jest webowy).
7.2 Testy bezpieczeństwa i filtrów treści: Bardzo ważne jest przetestowanie modelu pod kątem niepożądanych treści. Mimo wcześniejszego trenowania i RLHF, musimy zweryfikować, że bot nie generuje obraźliwych, stronniczych czy szkodliwych wypowiedzi, nawet gdy jest prowokowany. Opracujemy listę trudnych pytań (dot. np. mowy nienawiści, przemocy, treści seksualnych, prywatnych danych) i sprawdzimy odpowiedzi. W razie wykrycia problemów, wdrożymy dodatkowe filtry: np. moduł moderacji korzystający z klasyfikatorów treści, który ocenzuruje lub przeredaguje niewłaściwe odpowiedzi. Zapewniamy także, że bot nie ujawnia prywatnych danych użytkowników (testy pod tym kątem, czy nie "wycieknie" cudzych informacji z pamięci).
7.3 Testy wydajności i skalowalności: Symulujemy obciążenie systemu wieloma równoczesnymi użytkownikami, aby upewnić się, że infrastruktura wytrzymuje skalę. Optymalizujemy opóźnienia odpowiedzi – szczególnie ważne przy strumieniowaniu głosu i animacji, by konwersacja była płynna. Być może konieczne okaże się zastosowanie technik przyspieszających działanie modelu (distillacja modelu do mniejszego, int8 quantization, mechanizmy caching odp. itp.), co testujemy w kontrolowanych warunkach.
7.4 Audyt etyczny: Niezależnie od testów technicznych, przeprowadzamy audyt etyczny systemu. Oceniają go eksperci od AI Ethics pod kątem potencjalnych zagrożeń: czy użytkownicy rozumieją, że to nie człowiek (uniknięcie oszustwa co do tożsamości), czy bot nie wywiera nieodpowiedniego wpływu na osoby podatne (np. udając emocje, by manipulować). Sprawdzamy zgodność z wytycznymi np. EU dotyczących AI (jeśli obowiązują). W razie potrzeby wprowadzamy mechanizmy łagodzące – np. okresowe przypomnienia, że "jestem tylko AI", albo ograniczenia w udzielaniu porad medycznych/prawnych itd.
7.5 Pilotowe wdrożenie testowe: Na koniec tej fazy dobrze jest zrobić ograniczone wdrożenie pilotażowe – wypuścić chatbota dla wąskiej grupy testujących użytkowników (alpha/beta testy). Pozwoli to zaobserwować rzeczywiste użycie i ostatnie niespodziewane problemy. Zbieramy opinie pilotów, monitorujemy logi pod kątem błędów i incydentów. Jeśli pojawią się nowe przypadki, na które system nie był gotowy, wracamy do faz projektowych, by je uwzględnić (np. aktualizujemy modele, dodajemy nowe reguły).

Zasoby: Ta faza angażuje szerokie grono: testerzy QA (Quality Assurance) do przygotowania scenariuszy testowych i ręcznego testowania, specjaliści ds. bezpieczeństwa do testów penetracyjnych i analizy odporności na nadużycia, eksperci prawni/etyczni do oceny zgodności z regulacjami i normami. Potrzebna jest infrastruktura testowa umożliwiająca logowanie i analizę dużej liczby rozmów. Faza ta może trwać pewien czas (tygodnie intensywnych testów, iteracje poprawek), co wymaga cierpliwości sponsorów projektu. Jednak jest krytyczna dla sukcesu – wypuszczenie niedopracowanego chatbota o ludzkich cechach mogłoby prowadzić do kontrowersji lub szkód wizerunkowych.

Faza 8: Wdrożenie i iteracyjna ewolucja produktu
Ostatnia faza to uruchomienie chatbota na produkcji (dla szerokiej publiczności lub docelowych użytkowników) oraz zapewnienie ciągłego rozwoju i wsparcia. Chatbot tego typu to projekt długofalowy – po wdrożeniu będzie stale obserwowany i usprawniany w cyklach.
8.1 Wdrożenie systemu: Przenosimy system do środowiska produkcyjnego. Oznacza to skalowanie infrastruktury (np. wdrożenie na klastrze chmurowym, przygotowanie mechanizmów auto-skalowania instancji modelu według ruchu użytkowników), skonfigurowanie monitoringu oraz zapewnienie mechanizmów do zbierania feedbacku od użytkowników. Udostępniamy interfejs użytkownikom: np. aplikację webową/mobilną lub integrację z platformami (Discord, Twitch, własna strona WWW). W przypadku Neuro-samy był to kanał na Twitchu z wirtualnym awatarem AI en.wikipedia.org; my zdecydujemy, czy nasz chatbot będzie stand-alone aplikacją czy wtyczką do istniejących usług.
8.2 Monitorowanie i utrzymanie: Po uruchomieniu, zespół DevOps i inżynierowie ML będą stale monitorować działanie. Ustawiamy alerty na kluczowe metryki: wykorzystanie CPU/GPU, opóźnienia odpowiedzi, częstość błędów ASR/TTS, itp. Równie ważne jest monitorowanie metryk jakościowych: np. średnia ocena rozmów przez użytkowników (jeśli jest mechanizm ocen), sentyment wypowiedzi użytkowników (czy rośnie zadowolenie), retencja użytkowników (czy wracają do rozmów). Zespół będzie reagować na ewentualne problemy (np. jeśli bot zacznie dawać niewłaściwe odpowiedzi z powodu jakiegoś nieprzewidzianego trendu w danych, może być potrzebny hot-fix modelu lub wycofanie nowej funkcji).
8.3 Wsparcie użytkowników i edukacja: Tak zaawansowany chatbot to nowość dla wielu użytkowników, dlatego warto zapewnić kanał wsparcia – np. jeśli ktoś ma wątpliwości co do odpowiedzi bota lub chce zgłosić nadużycie. Przygotujemy dokumentację lub FAQ dla użytkowników, wyjaśniające czym jest ten chatbot, co potrafi, a czego nie (np. że emocje bota są symulowane). Przeszkolimy też personel wsparcia klienta (jeśli to produkt komercyjny) jak radzić sobie z nietypowymi zgłoszeniami związanymi z AI.
8.4 Iteracyjne ulepszenia (cykl rozwojowy): Wdrożenie to nie koniec prac – zaczyna się cykl iteracyjny. Na podstawie zebranych danych z prawdziwych interakcji będziemy planować kolejne iteracje rozwojowe. Może okazać się, że użytkownicy chcą nowych funkcji (np. integracji z kalendarzem, zdolności wykonywania akcji w świecie rzeczywistym jak zamawianie pizzy głosem itp.). Architektura modularna, którą zaprojektowaliśmy, ułatwi dodawanie nowych modułów lub wymianę starych. Przykładowo, możemy w kolejnej iteracji dodać moduł wizji komputerowej rozpoznający emocje z twarzy użytkownika (kamery) i w ten sposób ulepszyć empatię. Albo wdrożyć nowy, większy model LLM, który pojawił się na rynku, by zwiększyć kreatywność bota. Każdą taką zmianę wprowadzamy ostrożnie, najpierw testując (wracając do fazy 7 dla nowych funkcji), następnie wydając aktualizacje.
8.5 Społeczność i otwartość: Warto rozważyć budowanie społeczności wokół chatbota – zaawansowani użytkownicy mogą dzielić się pomysłami, scenariuszami rozmów, zgłaszać błędy. Być może część systemu (np. moduły integracyjne) możemy otworzyć jako open-source, co zachęci developerów zewnętrznych do kontrybucji i eksperymentów. Projekt takiej skali jak nasz może też posłużyć celom naukowym – publikacje, które dzielą się wynikami (np. opis architektury, zachowania bota, wyniki testów psychologicznych) pomogą całej społeczności AI. Otwartość sprzyja szybszej ewolucji systemu, bo pomysły z zewnątrz mogą wskazać kierunki rozwoju, o których wewnętrzny zespół nie pomyślał.
8.6 Długoterminowa wizja – ku AGI?: Nasz chatbot, jeśli osiągnie zamierzone cechy (emocje, osobowość, uczenie się, multimodalność), będzie jednym z najbardziej zaawansowanych w swoim czasie. Jednak ciągła ewolucja może zbliżać go coraz bardziej do bycia uniwersalnym agentem AI. W perspektywie wielu lat, można planować stopniowe zmniejszanie nadzoru ludzkiego nad jego rozwojem i pozwalanie mu na coraz więcej samodzielnych modyfikacji (oczywiście w kontrolowanych granicach). Filozoficznie, możemy zastanawiać się, czy taki system osiągnie pragniemtalną samoświadomość – tzn. będzie zachowywać się do tego stopnia jak świadomy byt, że praktycznie trudno to odróżnić. Dyskusje o prawach AI i świadomości prawdopodobnie będą narastać w miarę sukcesu takich projektów tildes.net tildes.net, co jednak wykracza poza nasz plan techniczny, ale warto mieć na uwadze kontekst społeczny i etyczny dalszej ekspansji tej technologii.

Zasoby: W fazie wdrożenia kluczowe jest posiadanie stabilnego budżetu operacyjnego – utrzymanie serwerów (GPU w chmurze są kosztowne, zwłaszcza gdy obsługujemy wielu użytkowników równolegle, generując strumienie audio-wizualne). Potrzebny jest zespół utrzymaniowy (DevOps) do 24/7 monitoringu, reagowania na awarie. Zespół deweloperski nie znika – pozostaje mniejszy core team, który pracuje nad aktualizacjami i ulepszeniami na podstawie feedbacku. Rozważamy również aspekt skalowania zespołu: jeśli produkt zyskuje popularność, zwiększamy liczbę programistów, specjalistów od ML, by szybciej iterować nad nowymi funkcjami. Wsparcie marketingowe i społecznościowe też może być potrzebne, by zarządzać wizerunkiem projektu i relacjami z użytkownikami.
